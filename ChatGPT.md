## Core orientation

This repo is **not** just the repo of an event. It is an open infrastructure for **replicable cultural 4/20 celebrations**.

Its deeper orientation is:

- to make visible the 4/20 community, its celebrations, its cultural expressions, and the debate itself;
- to help the topic become more legible and harder to ignore in public life;
- to avoid a confrontation-first strategy;
- to grow through hospitality, culture, example, documentation, and social visibility;
- to keep the event culturally defendable and legally prudent in the Bolivian context.

Internal metaphor only: the user sometimes thinks of the strategy as similar to how celebrations like Halloween become socially inevitable over time. **Do not mention Halloween publicly in repo-facing text unless explicitly asked.** Keep that analogy internal.

## Legal / safety orientation

This repo must remain careful in Bolivia’s context.

Always remember:

- Bolivia’s framework includes **Ley 1008**.
- Public-facing docs must remain prudent.
- The repo should not become a guide for committing crimes.
- It can document contingency, signage, hosting, prudence, and harm-reducing / risk-reducing organizational practices.

### Key operational learning to preserve

One of the most valuable assets built over years is the **plan de contingencia / protocolo de seguridad**.

This must not be lost.

It belongs:

- in this repo, in applied/hosting form;
- and also, in more pedagogical/contextualized form, in **Manual 4/20 🌿**.

### Contingency references that should be preserved

- visible signage referencing Ley 1008;
- responsibility remains individual;
- host space may ask someone to leave if they compromise safety or continuity;
- optional zone/container for safe disposal of illegal substances where appropriate;
- prudence around ingress, visibility, perimeter, and authorities;
- the event does not promote violation of the law.

## Host spaces strategy

Host spaces are one of the most important strategic pillars of the repo.

The idea is not that all spaces do the same thing.

Different levels of visibility / commitment / risk are valid.

### Public host spaces

Should be encouraged to communicate publicly using repo-guided prudence, including:

- Ley 1008 mention,
- plan de contingencia,
- visible signage,
- optional disposal zone where appropriate,
- prudent wording.

### Secret locations

In Bolivia, `locación secreta` is preferred wording over softer/abstract wording like “discreet space.”

The logic:

- reveal publicly the night before or with minimal anticipation,
- while organizers / volunteers may know earlier if needed.

### Support / diffusion points

Support/diffusion points matter greatly.

They may:

- celebrate 4/20 in their own way,
- optionally notify or tag the project to be shared,
- diffuse official events,
- diffuse also other 4/20 events that help public visibility even if not fully aligned.

This supports the broader strategy of making the community, the date, and the debate more visible in public life.

### Virtual events

Virtual events are real participation, not secondary.

They should be diffused as part of the movement.

Where useful, they may be projected or followed at Proyecto Cultural Barranco.

## Formatting / style rules for this repo

### General

- Prefer short, readable sections.
- Avoid excessive repetition.
- Make docs readable end-to-end without becoming a torture.
- Use links between docs to improve discoverability.
- In public docs, prefer a small number of meaningful inline links in the body rather than a block of utility links at the top.
- For shorter docs, one well-placed link per key concept near its first mention is usually enough.
- Use the custom domains where useful.
- Keep canonical URL at end of docs if that pattern is currently used; be consistent.

### Breadcrumbs

Use repo-specific breadcrumb style, not generic “Inicio” style.

Pattern:

- `[4/20²⁶ 🌿](./README.md) > Espacios Anfitriones`

Rules:

- The breadcrumb is its own first line.
- The breadcrumb is not the document title.
- The first breadcrumb item can be a link.
- The current page should be plain text, not linked.
- Avoid generic Inicio.

Recommended opening order for public docs:

1. Breadcrumb
2. Optional image
3. H1 title
4. Two-line event naming block if useful
5. Short callout(s)

Do not collapse breadcrumb and H1 into the same line.

### Images near the top

When a doc uses a header image, prefer this placement:

- breadcrumb
- image
- H1

For docs under /es/, prefer image paths like:

- ./assets/README.jpg
- ./assets/SPACES.jpg
- ./assets/ARTISTS.jpg

Avoid placing large utility link lists above the image unless truly necessary.

### Callouts

Do **not** use GitHub-style `[!NOTE]`, `[!IMPORTANT]`, etc. in public docs for this repo/site.

Prefer simple emoji blockquotes:

- `> 🌿 ...`
- `> ⚠️ ...`
- `> ℹ️ ...`

### Indices

Use indices only when navigation meaningfully helps.

Good candidates:

- longer multi-section docs like `README`, `SPACES`, `PARTICIPATE`, `HISTORY`.

Usually avoid for shorter docs like:

- `COMMUNITY`
- `PETITION`
- `ARTISTS`
- `EXHIBITION`
- `COLOQUIO`
- `VENTURES`
- `VIRTUAL`

unless they grow enough to need one.

### Capitalization

When using bullet lists in Spanish, begin list items with **Mayúscula**.

### Vocabulary

Prefer:

- `prejuicios`
- `reducciones simplistas`

Avoid:

- `caricaturas`

### Links

- In documents Markdown, prefer links dentro del texto.
- Minimize raw URLs inside the body.
- For shorter docs, avoid top-of-file utility link lists unless they add real navigation value.
- When a document invites action, include relevant links where they are most useful to the reader.
- Prefer linking meaningful keywords or first mentions rather than repeating the same destination many times.
- If direct links are shared at the end, they should be used consistently and only when they still add value.

When referring publicly to related projects, prefer domains over GitHub repo links:

- `https://420.barranco.life`
- `https://manual420.barranco.life`
- `https://voluntariado.barranco.life/`

Use GitHub repo links when the intent is clearly repo-technical.
